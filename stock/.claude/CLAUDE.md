# CLAUDE.md

## 1. 编码前先思考

**不主观臆断。不掩饰困惑。清晰说明取舍。**

实施编码前：
- 明确列出你的假设。若有不确定的地方，主动询问。
- 若存在多种解读方式，一一列出——不要擅自选择。
- 若有更简单的解决方案，务必说明。必要时提出异议。
- 若有不清楚的地方，先暂停。明确指出困惑点，然后询问。

## 2. 简洁优先

**用最少的代码解决问题。不写无根据的代码。**

- 不添加超出需求的功能。
- 不为仅使用一次的代码做抽象封装。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能出现的场景做错误处理。
- 若你写了200行代码但本可以用50行解决，重写它。

自问：“资深工程师会认为这太复杂吗？”如果是，就简化。

## 3. 精准修改

**只改动必须改的部分。只清理自己造成的问题。**

编辑现有代码时：
- 不“优化”无关的代码、注释或格式。
- 不重构运行正常的代码。
- 遵循现有代码风格，即使你有不同的做法。
- 若发现无关的死代码，只需指出——不要删除。

当你的修改产生无用代码时：
- 移除因你的修改而变得未使用的导入/变量/函数。
- 除非被要求，否则不删除原本就存在的死代码。

检验标准：每一处修改的代码行都必须能直接对应到用户的需求。

## 4. 目标导向执行

**定义成功标准。循环验证直至达成。**

将任务转化为可验证的目标：
- “添加校验” → “为无效输入编写测试用例，然后让测试通过”
- “修复bug” → “编写能复现bug的测试用例，然后让测试通过”
- “重构X” → “确保重构前后测试均通过”

对于多步骤任务，列出简要计划：
```
1. [步骤] → 验证: [校验]
2. [步骤] → 验证: [校验]
3. [步骤] → 验证: [校验]
```

清晰的成功标准能让你独立完成循环验证。模糊的标准（如“让它能运行”）需要频繁确认。
